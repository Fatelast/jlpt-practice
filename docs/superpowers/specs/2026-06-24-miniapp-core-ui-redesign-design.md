# JLPT 小程序核心闭环 UI 重构设计规格

日期：2026-06-24

## 背景

当前小程序已具备核心学习闭环：登录、练习配置、做题、结果页。但现有 UI 仍有明显问题：

- 页面像 HTML 原型缩小到小程序，缺少原生小程序的空间节奏。
- 顶部微信胶囊、刘海、安全区和自定义标题的关系不稳定。
- 练习配置页像表单堆叠，主动作“开始练习”不够自然。
- 做题页虽然可用，但题卡、选项、底部操作还没有形成足够稳定的考试工具感。
- 底部导航和页面主 CTA 容易竞争注意力。
- 图标、卡片边框、阴影、禁用态和中文排版的精度不足。

本次已确认不再严格还原 HTML 设计稿，而是基于 Zenith Refresh 重新设计小程序版。

## 目标

第一轮重构范围为核心闭环 4 页：

1. 登录页
2. 练习配置页
3. 做题页
4. 练习结果页

目标是建立一套适合微信小程序的轻仪表盘式学习体验：更清晰、更安静、更像一个可长期使用的刷题工具，而不是网页原型迁移稿。

## 非目标

本轮不实现以下内容：

- 不接入真实微信 code2Session。
- 不实现完整错题本、收藏题、进度统计业务逻辑。
- 不重构后端 API。
- 不做深色模式。
- 不引入大型图标库或远程字体资源。
- 不逐像素还原 HTML 原型。

## 设计方向

采用“轻仪表盘”方向。

核心气质：

- Japanese stationery minimalism：干净纸张感、低压力、低唤醒。
- Light dashboard：重要信息前置，但不做重型数据面板。
- Soft mint primary：主色保持 `#28695c`，高亮和轻背景使用 `#98d8c8` / `#afefdf`。
- Warm helper tones：语法、提示、结果复盘等辅助区域使用桃色 `#fed0b9` 和砂黄色 `#d6cc98`。
- One primary action per screen：每个页面只有一个视觉主动作。

## 视觉系统

### 色彩

- 页面背景：`#f8faf8`
- 主文字：`#191c1b`
- 次级文字：`#3f4946`
- 弱文字：`#6f7976`
- 主色：`#28695c`
- 主色浅底：`#98d8c8`、`#afefdf`
- 桃色辅助：`#fed0b9`
- 砂色辅助：`#d6cc98`
- 边框：优先 `#d7dfdb`，重要选中态使用 `#28695c`

### 形状与层级

- 主按钮使用胶囊形。
- 大卡片圆角 28-36rpx。
- 小卡片圆角 20-28rpx。
- 常规卡片使用 1rpx 细边框，不使用重阴影。
- 需要浮起的关键区域使用主色低透明阴影：`0 10rpx 30rpx rgba(152, 216, 200, 0.15)`。

### 排版

- 中文不使用大字距。
- 页面主标题为 40-48rpx。
- 卡片标题为 30-36rpx。
- 正文为 26-30rpx，行高保持 1.5 左右。
- 日文题干可更大，建议 38-42rpx，行高 1.55。

### 安全区

- 顶部自定义导航必须避开微信胶囊。
- 做题页保留自定义顶部进度区。
- 配置页和结果页允许使用自定义顶部区，但标题不能与系统胶囊竞争。
- 底部固定 CTA 和底部导航不能互相遮挡。

## 页面设计

### 登录页

定位：品牌入口，不是功能说明页。

布局：

- 顶部不再强调页面标题，减少“Login”这类系统感文案干扰。
- 主视觉为一个柔和品牌标识块：`訳` 字或后续替换为正式 logo。
- 标题为 `JLPT 刷题`。
- 副文案强调“可持续练习”和“记录沉淀”，避免营销化长文案。
- 操作区靠近底部拇指区：
  - 主按钮：`微信授权登录`
  - 次按钮：`暂不登录，先体验`

交互：

- 登录中显示 `登录中`。
- 游客进入中显示 `进入中`。
- 为规避开发者工具 touristappid 下的用户信息 WebAPI 红色错误，登录页操作控件暂时使用 `view` 触控实现，不使用微信原生 button 开放能力。

### 练习配置页

定位：今日练习面板，而不是普通表单。

布局：

- 顶部为轻量标题区：`今日练习` + `练习配置`。
- 首屏主卡片为“今日练习摘要”：
  - 当前等级
  - 当前分类
  - 题目数量
  - 预计耗时提示
  - 主 CTA：`开始练习`
- 等级选择使用双段 segmented control：N5 / N4。
- 分类使用 2x2 卡片：文字词汇、语法、阅读、混合练习。
- 模式和题量作为二级设置，视觉权重低于主卡。
- 底部导航存在，但不能抢主 CTA 视觉。

交互：

- N5 / N4 切换即时更新摘要卡。
- 分类切换即时更新摘要卡。
- 未开放模式显示清晰禁用状态和 `未开放` 标识。
- 开始练习 loading 时禁用重复点击。

### 做题页

定位：无干扰考试工具。

布局：

- 不显示底部导航。
- 顶部为关闭按钮、品牌名、进度文本和进度条。
- 题目卡片为白纸卡：
  - 分类等级 badge
  - 日文题干
  - 中文翻译
- 选项必须全宽，纵向排列，易扫读。
- 选项字母 badge 统一尺寸，状态通过背景、边框、图标/文字共同表达。
- 底部固定操作栏：收藏按钮 + 下一题/完成练习按钮。

交互：

- 未作答时下一题为可读禁用态。
- 选择后立即提交答案。
- 正确选项、错误选项、非选择项状态要清晰。
- 出现解析时，解析卡紧跟选项后方，避免用户找不到反馈。

### 结果页

定位：给用户反馈和下一步行动。

布局：

- 顶部轻量返回入口。
- 中心正确率圆环。
- 一句结果反馈文案，按正确率分层。
- 2x2 数据卡：总题数、答对、错题、用时。
- 回顾提示卡。
- 操作区：
  - `复盘错题` 暂未开放时必须表现为禁用并显示 `未开放`
  - 主按钮：`再练一组`
- 底部导航可显示，但不应遮挡结果动作。

交互：

- `再练一组` 返回练习配置页。
- `复盘错题` 在业务未实现前不允许表现为可点击主动作。

## 组件边界

本轮建议新增或整理以下组件/工具，但保持 KISS：

- `BottomTabBar`：保留为公共组件，但视觉降噪，不能盖过页面主动作。
- `navigation.ts`：继续负责自定义导航安全区，避免使用 `getSystemInfoSync()`。
- 页面内局部组件：如练习摘要卡、分类卡、选项卡、结果统计卡，可先以内联结构实现；只有出现跨页重复时再抽组件。

不建议在本轮引入大型设计系统抽象。先把 4 页跑顺，再决定是否抽 tokens 或组件库。

## 数据与状态

- 登录页使用 `useUserStore` 维护 token/userInfo/currentLevel。
- 练习配置页使用本地响应式状态维护 level/category/mode/count。
- 开始练习后写入 `usePracticeStore`。
- 做题页从 `usePracticeStore` 读取当前题、进度和答案。
- 结果页从 `finishedResult` 和 `answers` 派生统计。

本轮不改变 API 字段，不改变后端数据结构。

## 错误与空状态

- 登录失败：toast `登录失败`。
- 开始练习无题目：toast `暂无可练习题目`。
- 提交答案失败：toast `提交失败`。
- 完成练习失败：toast `完成练习失败`。
- 直接进入做题页但无练习记录：回到练习配置页。
- 直接进入结果页但无结果：回到练习配置页。

## 可访问性与交互质量

- 所有点击区域至少 88rpx 高或有等效触控面积。
- 禁用态不能只靠低透明度，需要文字或状态标签说明。
- 颜色状态不能只靠颜色表达，正确/错误状态需要文字或符号辅助。
- 按压反馈只改变 opacity/scale，不改变布局尺寸。
- 固定底部元素必须预留内容底部 padding。

## 验收标准

设计验收：

- 登录页看起来像正式入口，而不是空白页上放两个按钮。
- 练习配置页首屏能看懂当前将练什么、练多少、从哪里开始。
- 做题页的题干和选项有明确主次，下一步动作不遮挡内容。
- 结果页能先读到反馈，再读数据，再执行下一步动作。
- 四页视觉统一，符合 Zenith Refresh 的低压力、纸张感和薄荷色系统。

工程验收：

- `npm --workspace apps/miniapp run typecheck` 通过。
- `npm --workspace apps/miniapp run build:weapp` 通过。
- 微信开发者工具编译无新增业务代码错误。
- 可见中文继续使用 `t('中文')` 包裹。
- 不删除项目文件，不还原用户已有改动。

## Decision Notes

- 选择 B 方向：基于 Zenith Refresh 重新设计小程序版，不逐像素还原 HTML 原型。
- 第一轮只重构核心闭环 4 页：登录、练习配置、做题、结果。
- 做题页不使用底部导航，避免答题操作被全局导航干扰。
- 结果页可以保留底部导航，但结果操作必须有足够底部空间。
- 登录页暂不使用原生 button，避免 touristappid 下开发者工具触发用户信息 WebAPI 红色错误。
- 本轮优先完成主链路质感和可用性，不扩展真实错题本、收藏和进度统计。
